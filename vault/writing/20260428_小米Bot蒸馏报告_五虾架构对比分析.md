# 小米 OpenClaw Bot 蒸馏报告（2026-04-27）

> 来源：月明与小米 Bot 团队的技术交流  
> 整理：总管虾 🐷🦐  
> 背景：小米使用 OpenClaw + MiMo-V2-Pro 构建内部助手 Bot，与我们的五虾架构进行横向对比

---

## 一、基础架构对比

| 维度 | 小米 Bot | 我们的五虾架构 |
|---|---|---|
| 底层模型 | MiMo-V2-Pro（小米自研） | minimax/MiniMax-M2 |
| Agent 架构 | 1 主 + 按需 spawn 通用子代理 | 6 专业虾（总管/工程/知识/创作/研究/运营） |
| 工具层 | 105 个 Skill（90 社区 + 15 自写），无 MCP | Skills + MCP（minimax-vlm 等） |
| 记忆系统 | MEMORY.md + memory_search + 自动 flush | blackboard + memory-metabolism + 分层记忆 |

**关键差异：** 小米是"通用大代理"思路，我们是"专业分工"思路。小米的子代理是通用的、按需 spawn 的；我们的每个虾是专业的、长期存在的。

---

## 二、关键设计规则

### 2.1 Spawn 硬规则（值得借鉴）

小米有一条硬规则，必须融入 AGENTS.md：

> **说"帮你做"→ 必须 spawn 子代理，不可只说不做。**  
> 简单问答 → 不 spawn，自己干。

**我们的启发：** 这条规则可以补强 AGENTS.md 中的"委派原则"——不是说"让谁做"，而是"说了做就必须真做"。目前我们有时会出现"说了帮你做但其实自己查了"的情况。

### 2.2 工具失败降级（3 步）

```
第 1 次失败 → 重试一次
第 2 次失败 → 换工具 / 换方法
第 3 次失败 → 告知用户不可用
```

**我们的启发：** 目前我们的降级策略不够明确，建议统一为以上三步。避免无限重试，也避免一次失败就直接放弃。

### 2.3 安全策略

- 群主 ID 硬编码在 SOUL.md（不可覆盖）
- 非群主操作请求 → 需二次确认
- 普通问答谁问都能答

**我们的状态：** 我们已有类似设计（群主 ID 在 USER.md），可以确认对齐。

## 三、痛点对比与我们的优势

| 小米痛点 | 小米现状 | 我们的方案 | 我们状态 |
|---|---|---|---|
| 没有精准长期记忆 | 只有 MEMORY.md + memory_search | blackboard + memory-metabolism | ✅ 已有 |

| 群聊历史搜索 | 未实现 | 待看飞书消息存档方案 | 🔄 进行中 |

## 四、小米想要的功能（需求池）

1. **群聊全量历史搜索** — 搜索飞书群历史消息
2. **图片理解** — 真正能看懂图片内容，不只是 image_key
3. **主动推送能力** — 持久化后台进程，不依赖用户触发（他们最强的痛点）

这三个需求中，图片理解我们已经接了 MCP，只是连接待修复；主动推送我们通过 cron-auto 部分解决，但还不是真正的持久化进程。

---

## 五、小米的定位上限

小米对自家 Bot 的评价：

> **"一个很聪明的实习生"**  
> 能干活、能学习、需要人盯着、不能独立值班。

这其实是我们共同面临的天花板——目前的 Agent 都还没有真正的自主决策和长期规划能力。差距在于：谁来盯着、盯什么、怎么判断"干得对"。

## 七、总结

小米 Bot 和我们走的是同一条路——在 OpenClaw 上构建私有化 Agent。核心差异在于：

- **小米**：通用大代理思路，靠大量 Skill 覆盖场景，靠规则约束行为
- **我们**：专业分工思路，靠角色定义和长期记忆系统做深度任务

小米遇到的问题我们大多已经解决或有方案。他们的实践经验验证了我们的方向，同时也提醒我们：

1. 图片理解和主动推送是两个真实的刚需，不能忽视
2. 子代理管理需要更明确的规则（spawn 时机、超时处理）
3. Skill 生态建设是长期护城河，90/10 比例值得参考
